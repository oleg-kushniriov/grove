◆ Shared Cluster vs Per-Suite Configuration

All tests run against the same physical cluster and reuse the same *k8s.Clients instance. The suite is a per-test view: "give me N nodes, this workload, these timeouts, and fresh manager wrappers."

Aspect Scope Details
Cluster shared Single cluster for the entire test run
K8s Clients shared One *k8s.Clients instance created in TestMain
Available Nodes per-suite requiredWorkerNodes arg controls how many nodes are uncordoned
Workload per-suite Each test provides its own WorkloadConfig (YAML path, expected pods, name)
Timeouts per-suite Optionally overridden via WithTimeout() / WithInterval() (e.g. scale tests: 15m vs default 4m)
Manager Instances per-suite Fresh structs per suite, but all point to the shared *k8s.Clients
Diagnostics per-suite Fresh DiagCollector scoped to the test name for output files
Resource State per-suite cleanup() calls CleanupWorkloads() after each test — each suite starts clean

◆ LegacyTestContext vs TestContext

The refactor changed how tests consume the shared cluster and clients. The cluster lifecycle itself (SharedClusterManager) is identical in both paths.

1. Client Ownership: Scattered Fields vs Bundled Struct
legacy — LegacyTestContext
type LegacyTestContext struct {
  Clientset      kubernetes.Interface
  DynamicClient  dynamic.Interface
  RestMapper     meta.RESTMapper
  RestConfig     *rest.Config
  CRClient       client.Client
  // + Namespace, Timeout, etc.
}
5 separate client fields on the struct. Also requires a clientCollection intermediary to unpack *k8s.Clients back into individual fields.
new — TestContext
type TestContext struct {
  Clients  *k8s.Clients // all 5 in one
  Pods     *PodManager
  Nodes    *NodeManager
  Resources *ResourceManager
  // + Grove managers, Diag, etc.
}
Single *k8s.Clients bundle passed to managers at construction. No unpacking needed.
2. Operations: Wrapper Functions vs Domain Managers
legacy
// thin wrapper → standalone util func
func (tc LegacyTestContext) listPods() {
  return utils.ListPods(
    tc.Ctx, tc.Clientset,
    tc.Namespace, ...)
}

func (tc LegacyTestContext) cordonNode(n) {
  return utils.SetNodeSchedulable(
    tc.Ctx, tc.Clientset, n, false)
}
Every method manually threads the right client. Logic lives in utils package as free functions.
new
// delegates to manager (client inside)
func (tc *TestContext) ListPods() {
  return tc.Pods.List(
    tc.Ctx, tc.Namespace, ...)
}

func (tc *TestContext) CordonNode(n) {
  return tc.Nodes.Cordon(
    tc.Ctx, n)
}
Managers encapsulate the client reference. No need to pass it per call.
3. Diagnostics: Inline Methods vs Dedicated DiagCollector
legacy
// methods on LegacyTestContext itself
// in debug_utils.go (~400 lines)

tc.CollectAllDiagnostics()
tc.dumpOperatorLogs(logger)
tc.dumpGroveResources(logger)
tc.dumpPodDetails(logger)
tc.dumpRecentEvents(logger)
Diagnostics methods live directly on LegacyTestContext. Uses tc.Clientset and tc.DynamicClient inline. File output handled within the methods.
new
// separate struct in diagnostics/

tc.Diag = diagnostics.NewDiagCollector(
  clients, ns, mode, dir, logger)

tc.Diag.CollectAll(t.Name())
Dedicated *DiagCollector struct in its own package. Initialized once per suite with all config. Single method to trigger collection.
4. Cleanup: Manual Wiring vs Consistent Pattern
legacy setup.go:196-229
cleanup := func() {
  // manually create LegacyTestContext
  // JUST for diagnostics
  diagTc := LegacyTestContext{
    T: t,
    Clientset: clients.Clientset,
    DynamicClient: clients.DynamicClient,
    RestMapper: clients.RestMapper,
    Namespace: "default",
    DiagMode: diagMode,
    DiagDir: diagDir,
  }
  if t.Failed() {
    diagTc.CollectAllDiagnostics()
  }
  sharedCluster.CleanupWorkloads(ctx)
}
new context.go:134-148
cleanup := func() {
  if t.Failed() {
    tc.Diag.CollectAll(t.Name())
  }
  sharedCluster.CleanupWorkloads(ctx)
}
Uses the already-constructed test context. No manual struct assembly. DiagCollector was initialized in NewTestContext().
5. The auto-mnnvl Special Case
The auto-mnnvl/ tests have their own testContext (lowercase, different type in testutils.go) and their own prepareTestCluster() that returns raw clients individually:
clientset, restConfig, dynamicClient, groveClient, cleanup := prepareTestCluster(ctx, t, 0)
They don't use TestContext from setup.go at all — they call SharedCluster directly and pass raw clients around. This is a third variant that predates both abstractions.
6. What Stayed the Same
Both paths go through SharedClusterManager identically:
SharedCluster.PrepareForTest(ctx, N) cordon/uncordon nodes
SharedCluster.GetAllClients() same shared *k8s.Clients
SharedCluster.CleanupWorkloads(ctx) same resource cleanup
The cluster lifecycle and client creation are identical. The refactor only changed how tests consume those shared resources.
